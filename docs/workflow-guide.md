# 워크플로우 가이드

OMC, pm-skills, autoresearch를 조합해서 실제 업무에 적용하는 워크플로우를 정리한다. 각 케이스별로 어떤 순서로, 어떤 명령을 내리는지 구체적으로 안내한다.

---

## 1. 새 제품/기능 아이디어 → 출시까지

처음 아이디어만 있는 상태에서 검증, 기획, 구현, 출시까지 전 과정을 커버하는 풀사이클 워크플로우.

### Phase 1: 아이디어 검증

```
/brainstorm ideas 새로운 할일 관리 앱
```
→ 3관점(PM·디자이너·엔지니어) 아이디어 발산

```
/discover 할일 관리 앱 — 반복 업무 자동화에 집중
```
→ 가정 식별 → 우선순위 → 실험 설계까지 7단계 자동 진행

```
/interview prep 반복 업무가 많은 직장인, 30분 인터뷰
```
→ 인터뷰 스크립트 생성 (Mom Test 원칙 적용)

인터뷰 실시 후:
```
/interview summarize [인터뷰 녹취/메모 붙여넣기]
```
→ JTBD, 인사이트, 검증/반증된 가정 정리

### Phase 2: 전략 수립

```
/value-proposition 반복 업무 자동화 할일 관리 앱
```
→ JTBD 기반 6파트 가치 제안

```
/strategy 반복 업무 자동화 할일 관리 앱, B2C SaaS
```
→ 9섹션 전략 캔버스

```
/competitive-analysis 할일 관리 앱 시장 (Todoist, Notion, TickTick)
```
→ 경쟁 환경 분석, 포지셔닝 맵, 차별화 기회

```
/pricing B2C SaaS 할일 관리 앱, 프리미엄 모델 고려
```
→ 가격 모델, 티어 구조, 수익 예측

### Phase 3: 실행 기획

```
/write-prd 반복 업무 자동 감지 및 템플릿 생성 기능
```
→ 8섹션 PRD

```
/write-stories [PRD 내용 붙여넣기]
```
→ 스프린트 단위 유저 스토리 5~15개

```
/pre-mortem [PRD 내용 붙여넣기]
```
→ 출시 전 리스크 분석

```
/plan-okrs 할일 관리 앱 팀, Q2, 회사 목표: MAU 10만 달성
```
→ 분기 OKR 3세트

### Phase 4: 구현 (OMC)

```
architect 에이전트에게 PRD 기반 시스템 설계 검토 요청
```
→ 아키텍처 리뷰 (읽기 전용)

```
/team 3:executor "PRD 기반으로 반복 업무 자동 감지 기능 구현"
```
→ 3개 에이전트가 병렬로 구현

```
/ultraqa "모든 유저 스토리의 수용 기준 통과 확인"
```
→ 테스트 → 수정 → 재테스트 반복

### Phase 5: 출시

```
/plan-launch 반복 업무 자동화 할일 관리 앱, B2C, 한국 시장
```
→ 비치헤드, ICP, 채널 전략, 타임라인

```
/sprint release-notes [이번 스프린트 티켓/커밋 목록]
```
→ 사용자 대상 릴리즈 노트

## 2. 경쟁사 대응 전략

경쟁사가 새 기능을 출시했거나, 시장에 새로운 경쟁자가 등장한 상황.

```
/competitive-analysis [경쟁사 이름과 우리 제품 설명]
```
→ 기능 비교, 포지셔닝 맵, 위협/기회 분석

```
/market-scan [우리 제품이 속한 시장]
```
→ SWOT + PESTLE + Porter's + Ansoff 통합 환경 분석

```
/battlecard [우리 제품] vs [경쟁사], 대상: 엔터프라이즈 바이어
```
→ 세일즈팀용 배틀카드 (이의 처리, 전략 질문 포함)

```
/strategy [분석 결과를 반영한 전략 재수립]
```
→ 업데이트된 전략 캔버스

필요시 OMC로 빠른 기능 구현:
```
/autopilot "경쟁사 대응 — [구체적 기능] 구현"
```

## 3. 사용자 피드백 기반 개선

앱스토어 리뷰, NPS 응답, 서포트 티켓 등이 쌓인 상황.

```
/analyze-feedback [CSV 또는 피드백 텍스트 붙여넣기]
```
→ 감성 분석, 테마 추출, 세그먼트별 비교

```
/research-users [서베이 응답/인터뷰 노트 붙여넣기]
```
→ 페르소나 3~4개, 행동 세그먼트, 고객 여정 맵

```
/triage-requests [피드백에서 추출된 기능 요청 목록]
```
→ 테마별 우선순위 + 노력/정합성 평가

우선순위가 높은 항목에 대해:
```
/write-prd [최우선 기능 요청]
```
→ PRD 작성

```
/ralph "PRD 기반 구현 → 코드 리뷰 → 테스트 통과까지"
```
→ 구현~검증 자동 반복

## 4. A/B 테스트 사이클

가설 수립부터 실험 설계, 결과 분석까지.

### 가설 수립
```
/brainstorm experiments 온보딩 완료율 개선
```
→ 실험 아이디어 + 테스트 가능한 가설

### 지표 설정
```
/setup-metrics 온보딩 플로우, 핵심 지표: 7일 리텐션
```
→ 노스스타, 입력 지표, 알림 임계값

### 데이터 준비 (필요시)
```
/write-query 코호트별 온보딩 완료율과 7일 리텐션 상관관계
```
→ SQL 쿼리 생성

### 구현
```
/team 2:executor "온보딩 A/B 테스트 변형 구현 — A: 기존, B: 프로그레스바 추가"
```
→ 변형 구현

### 결과 분석
```
/analyze-test Control: 32% 완료율 (n=5000), Variant: 38% 완료율 (n=4800)
```
→ 통계적 유의성 + 비즈니스 임팩트 + SHIP/EXTEND/STOP 판정

```
/analyze-cohorts [실험 기간 유저 활동 데이터]
```
→ 코호트별 리텐션 비교

## 5. 분기 계획 수립

분기 시작 시 전략 → OKR → 로드맵 → 스프린트까지 정렬하는 워크플로우.

```
/strategy [제품 현황 + 지난 분기 성과 + 시장 변화]
```
→ 전략 캔버스 업데이트

```
/north-star [제품 설명 + 비즈니스 모델]
```
→ 노스스타 메트릭 검증/재설정

```
/plan-okrs [팀명], Q2, 회사 목표: [목표 목록]
```
→ 팀 OKR 3세트

```
/transform-roadmap [현재 기능 백로그]
```
→ 성과 중심 로드맵으로 전환

```
/stakeholder-map [분기 프로젝트 설명]
```
→ 이해관계자 관리 전략 + RACI

```
/sprint plan [첫 스프린트 스토리 + 팀 용량]
```
→ 스프린트 계획

## 6. 새 시장 진입

기존 제품으로 새로운 시장/세그먼트에 진입하는 상황.

```
/market-scan [진입 대상 시장]
```
→ SWOT + PESTLE + Porter's + Ansoff 통합 분석

```
/research-users [타겟 시장 리서치 데이터]
```
→ 새 시장의 페르소나, 세그먼트, 고객 여정

```
/business-model all [새 시장 맥락]
```
→ 4개 캔버스 동시 분석 + 교차 종합

```
/value-proposition [새 시장 타겟 유저 맥락]
```
→ 시장별 차별화된 가치 제안

```
/plan-launch [제품명], [새 시장], [제약 조건]
```
→ GTM 전략 (비치헤드, ICP, 채널, 타임라인)

```
/growth-strategy [제품 + 새 시장 맥락]
```
→ 성장 루프 + GTM 접근법 + 90일 실행 계획

```
/market-product [새 시장용 마케팅]
```
→ 캠페인, 포지셔닝, 카피, 제품명 제안

## 7. 스프린트 운영 루틴

매 스프린트 반복되는 일상 워크플로우.

### 스프린트 시작
```
/sprint plan [스토리 목록 + 팀 용량 + 리스크]
```
→ 용량 분석, 스토리 추천

### 스프린트 중 — 구현
```
/ultrawork "스토리 A, B, C 병렬 구현"
```
→ 독립 작업 병렬 실행

코드 리뷰:
```
code-reviewer 에이전트에게 PR 리뷰 요청
```

### 스프린트 종료
```
/sprint retro [팀 피드백 또는 스프린트 지표]
```
→ 회고 요약 + 액션 아이템

```
/sprint release-notes [완료된 티켓/커밋]
```
→ 사용자 대상 릴리즈 노트

### 회의 후
```
/meeting-notes [회의 녹취/메모]
```
→ 구조화된 회의록 + 결정사항 + 액션 아이템

## 8. 스킬 품질 개선

만든 스킬이 기대만큼 동작하지 않을 때 autoresearch로 자동 최적화.

```
/autoresearch
```

프롬프트에 따라 입력:
- **대상 스킬:** 최적화할 SKILL.md 경로
- **테스트 입력:** 3~5개 다양한 시나리오
- **Eval 기준:** 3~6개 binary 체크

예시:
```
대상: .claude/skills/my-reviewer/SKILL.md
테스트 입력:
  1. React 컴포넌트 PR
  2. API 엔드포인트 변경 PR
  3. DB 마이그레이션 PR
  4. 설정 파일 변경 PR
Eval:
  1. 보안 이슈를 정확히 식별했는가?
  2. 오탐(false positive)이 없는가?
  3. 수정 제안이 구체적인가?
  4. 심각도 분류가 적절한가?
```

→ 자율 실험 루프 실행 → 대시보드에서 진행 모니터링 → 최적화된 스킬 수령

```

```
/business-model all [현재 비즈니스 모델]
```
→ 4개 캔버스로 사업 구조 분석

```
/market-scan [시장 환경]
```
→ 시장 분석 (Porter's, PESTLE 등)

```
/competitive-analysis [경쟁 환경]
```
→ 포지셔닝 맵 + 차별화 전략

```
/north-star [핵심 지표 정의]
```
→ 노스스타 + 입력 지표 (투자자 대시보드용)

```

## 10. 디버깅 + 근본 원인 분석

프로덕션 이슈가 발생한 상황에서 OMC 에이전트를 활용한 체계적 디버깅.

### 단순 버그
```
debugger 에이전트에게 에러 로그와 함께 디버깅 요청
```
→ 5단계 근본 원인 분석

### 복잡한 버그 (원인 불명)
```
/trace "API 응답 시간이 간헐적으로 10초 이상 걸림"
```
→ 경쟁 가설 오케스트레이션, 증거 기반 추적

### 보안 이슈
```
security-reviewer 에이전트에게 해당 모듈 보안 감사 요청
```
→ OWASP Top 10 기반 취약점 탐지

### 수정 후 검증
```
/ultraqa "수정된 버그에 대한 회귀 테스트 + 관련 기능 테스트"
```
→ 테스트 통과까지 자동 반복

## 조합 원칙

1. **pm-skills로 "무엇을"** 정의하고, **OMC로 "어떻게"** 실행한다
2. **pm-skills는 순서대로** 체이닝하면 각 단계의 산출물이 다음 단계의 입력이 된다
3. **OMC 워크플로우는 상황에 맞게** 선택한다:
   - 단일 작업 → `executor` 에이전트 직접 호출
   - 독립된 다중 작업 → `/ultrawork` 또는 `/team`
   - 완료까지 반복 → `/ralph`
   - 완전 자율 → `/autopilot`
   - 품질 보증 → `/ultraqa`
4. **autoresearch는 메타 레벨** — 다른 스킬의 품질을 올리는 데 사용
5. **리서치와 실행을 분리** — pm-skills로 분석/기획 → OMC로 구현/검증
